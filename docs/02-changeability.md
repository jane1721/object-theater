# 📄 02-1 - 객체 중심 구조에서의 변경 용이성

### 왜 **가격 정책 변경**이 특정 객체만 수정하면 되는 건지?
**책임을 가진 객체**만 수정하면 되기 때문이다.

- 현재 구조에서는 가격(금액)과 관련된 정보는 `Ticket` 과 `TicketOffice`가 관리하고 있다.
  - `Ticket`: 티켓 1장의 가격(`fee`)을 가진 **값 객체**
  - `TicketOffice`: 판매할 티켓들의 목록과, 판매 금액(`amount`)을 관리하는 **역할 객체**

- 예시
  - 티켓 가격을 10,000원 → 12,000원 으로 인상하고 싶다면?
    - `Ticket` 객체에서 가격만 바꾸면 된다.
  - "특정 시간대에 50% 할인" 과 같은 정책을 추가하고 싶다면?
    - `TicketOffice` 가 티켓을 판매할 때 "할인"을 계산하도록 변경하면 된다.
  - → `Theater`, `Audience`, `TicketSeller` 등 다른 객체는 수정할 필요가 X

### 어떻게 이게 가능한건지?
객체들이 **자기 일만** 하고, 다른 객체의 **내부 상태**나 **세부 구현**을 몰라도 되기 때문이다.

- `Audience` 는 "티켓이 필요하다"고만 말한다.
- `TicketSeller` 는 "티켓을 팔아라"고만 요청 받는다.
- 실제 가격이 얼마인지, 할인받는지는 `TicketOffice` 가 알아서 처리한다.

### 정리

| **절차지향 구조**                  | **객체 협력 구조**                              |
|------------------------------|-------------------------------------------|
| 하나 수정하면 모든 관련 코드를 찾아야 함      | 수정해야 할 책임 객체 하나만 수정하면 됨                   |
| 하나 수정하면 모든 관련 코드를 찾아야 함      | 수정해야 할 책임 객체 하나만 수정하면 됨                   |
| 모든 클래스가 가격 정보를 알거나 직접 변경해야 함 | 가격은 `Ticket`, 판매 규칙은 `TicketOffice`가 책임 짐 |
| 확장/변경 시 버그 발생 가능성 높음         | 변경이 안전하고, 테스트하기 쉬움                        |

→ 책임을 명확히 가진 객체가 있기 때문에, 변경할 대 수정 범위가 작아진다.
